## Atomic-settle composition

The Pay.sh smoke composes the production atomic-settle path:

  1. `gate_payment_strict` — PolicyVault decision + state mutations
  2. SPL `TransferChecked` — settles the payment
  3. `emit_feedback` — CPI into Quantu

When `atomicityEnforced: true` on the SDK side and the production
`composeAtomicSettleTx` path is used, all three land in one signed
transaction. Per-instruction commit/revert atomicity is exercised by
`tests/policy-vault.spec.ts` "gate_payment_strict + set_killswitch
atomic" describe block (Phase F2.1) — flip-then-gate proves the gate's
revert rolls the killswitch flip back too, never half-way.

Live-devnet atomic settle composition is the load-bearing path of the
Pay.sh smoke above. The signed transfer + feedback emission share the
same payer / payee context (paymentIdHash binds them); the Phase D
Anchor + SDK commits installed the production composeAtomicSettleTx
helper and the `AtomicityEnforced` literal-type guard.

## Killswitch demo

Covered by `tests/policy-vault.spec.ts`:

  * "1-of-1 multisig can pause and unpause" — flip, observe Deny path,
    flip back, observe Allow.
  * "KillSwitch paused causes Deny — no state mutation" — fail-fast
    Deny without advancing spending counters.
  * "KillSwitch paused reverts the tx with KillSwitchEngaged"
    (gate_payment_strict) — full revert.
  * "killswitch flip before gate — gate reverts, flip rolled back"
    (atomic) — same-tx ordering proves the revert is total.

Live-devnet killswitch demo is also wired through Pay.sh via the
`/admin/pause` route in the demo Express app; the on-chain effect is
identical to the Anchor-test coverage. No separate JSON proof file —
the KillSwitchState PDA's `paused_at_slot` field is the on-chain
record.
