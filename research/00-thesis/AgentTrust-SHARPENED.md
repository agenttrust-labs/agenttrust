## Architecture — three components

### Component 1 — PolicyVault (the moat)

Anchor program implementing per-agent policy PDAs that read external Foundation-endorsed identity + reputation as inputs. The headline instruction `gate_payment(payer_agent, payee_agent, amount, mint, policy_id)` returns Allow/Deny/RequireValidation by composing 3 policy kinds in v1: **Spending** (per-tx + daily + weekly limits with rollover), **CounterpartyTier** (manual deserialization of `AtomStats.trust_tier` from atom-engine, gates on `min_counterparty_tier`), **KillSwitch** (multisig-controlled emergency pause). Velocity + RequireValidation policy kinds ship in v1.1. Light Kani harness proves 1-2 critical invariants (`paused → no Allow`, `velocity counter ≤ limit`). Manual cross-program PDA-deserialization pattern is the technical differentiator against asset-layer hooks (per Day-4 reverse-mapping problem) — PolicyVault sits at the agent-relationship layer where reputation gating is natural; AgentSafe Hooks sits at the mint layer where it isn't.

**What we consume from Foundation primitives:** `agent-registry-8004::AgentAccount` PDAs (identity), `atom-engine::AtomStats` PDAs (reputation tier + risk score). Both MIT-licensed, mainnet-deployed, Foundation-endorsed at solana.com/agent-registry.

**What we ship:** Anchor program (~800 LOC), 3 policy-kind modules, gate_payment composer, light FV harness, devnet integration tests against actual Quantu mainnet ATOM Engine.

**Moat status:** PRIMARY. The moat is here.

### Component 2 — TrustGate x402 (the demo-vehicle for PolicyVault)

Reference x402 facilitator (Anchor program + TypeScript service) that demonstrates PolicyVault in a live integration. **Pre-flight (POST /verify):** calls `policy_vault::gate_payment` via CPI; returns 402 + reason code if Deny, proceeds if Allow. **Settlement (POST /settle):** standard x402 token transfer (the mint can independently have AgentSafe Hooks enabled — defense-in-depth). **Post-settlement feedback emission:** facilitator-PDA-signed CPI to `agent_registry_8004::give_feedback` with score=100 (clean settlement) or lower (disputed/reverted). Closes the trust loop — today no in-protocol consumer of payment outcomes feeds back into ATOM reputation; TrustGate is the first programmatic feedback emitter.

**What we consume from Foundation primitives:** `agent-registry-8004::give_feedback` instruction (PDA-signed CPI surface).

**What we ship:** Anchor program (~300 LOC) + TypeScript Express service (~600 LOC). NO drop-in module for other facilitators in v1 (Phase-2 work).

**Moat status:** FEATURE. Demo-vehicle for PolicyVault. Without PolicyVault, TrustGate is a thin glue module facilitators could write themselves.

### Component 3 — ValidationRegistry (the standard-completion vehicle, ships as docs + stub PDA in v1)

Anchor program implementing ERC-8004's archived third leg — `request_validation` / `respond_to_validation` / `revoke_validation` / `read_attestation` for permissionless capability attestations. v1 SHIPS: `ValidationAttestation` PDA + `request_validation` instruction + `read_attestation` view helper. v1 DOES NOT SHIP: `respond_to_validation` flow (off-chain validator coordination), sybil-resistant attestor weighting (downstream-consumer-filtering pattern only). Docs explain the permissionless-attestor model + capability-namespace conventions + how PolicyVault's RequireValidation policy kind (v1.1) will consume attestations.

**What we consume from Foundation primitives:** None directly (this is the third leg the Foundation page references but Quantu archived).

**What we ship in v1:** Anchor program (~150 LOC stub), `ValidationAttestation` PDA, `request_validation` + `read_attestation`, full spec doc explaining v1.1+ roadmap.

**Moat status:** NARRATIVE-COMPLETION FEATURE. The "completes the ERC-8004 trinity" pitch beat depends on this existing in the repo as spec + stub, not on full productization in v1. Phase-2 work productizes.

## Moat — single paragraph

**PolicyVault is the structural moat.** Three mutually-reinforcing defensibility properties: (1) the *reverse-mapping problem* — `agent-registry-8004` exposes no public PDA at `["wallet_to_agent", wallet]`, so AgentSafe Hooks v1.1 (asset-layer) structurally cannot derive "which agent does this transfer represent" cheaply; AgentSafe could only add reputation-aware gating via architectural compromises that collapse into PolicyVault's design point; (2) the *Foundation-stack-completion* defense — SAEP's TreasuryStandard ships per-tx/daily/weekly limits but they're all agent-SELF-spending; SAEP would have to abandon their own AgentRegistry to consume Quantu's, which is an architectural reversal of their 10-program stack and gives them dual-registry confusion; by the time SAEP could pivot, AgentTrust holds reference-implementation status; (3) the *first-buyer-pull* defense against Foundation absorbing the slot into a future SDP module — shipping FIRST and getting the FIRST x402 facilitator integration (Dexter or atxp_ai) creates compounding switching costs (audit, integration, reference-implementation status) that make absorption expensive vs endorsement; Foundation historically prefers endorsement (they endorsed Quantu rather than building their own ERC-8004 port). The single risk that AgentSafe Hooks does NOT carry: Foundation/Quantu dependency-risk, mitigated by MIT-license-survivability (program continues regardless of Quantu corp status) + graceful-degradation mode (PolicyVault falls back to velocity-only policy if registry becomes unavailable).

## Founder narrative (in Mohit's voice)

I came to AgentTrust through a friend's startup. They were experimenting with AI agents that bought things on behalf of users — flights, gear, subscriptions. The first time it worked end-to-end, I watched an agent get hustled by a fraudulent storefront pretending to be a real brand — paid two thousand dollars for fake goods because there was no way for the agent to tell the difference. Same problem when you flip the other way: an agent paying an API for inference, getting billed by a scam wrapper pretending to be Anthropic. Real money, real fraud, no standard way to check. I spent a month and a half thinking about trust systems for agents — fake-review handling, sybil-resistance, gaming attacks on reputation. Then Quantu shipped the Solana Agent Registry with Foundation endorsement, and the ATOM Engine on top — identity and sybil-resistant reputation, two of three legs of ERC-8004. The third leg (validation) was archived "planned for future upgrade." That archived gap, plus the missing policy enforcement that gates payments on counterparty reputation, plus the missing closed-loop feedback that turns settlement outcomes back into reputation — that's AgentTrust. I'm not building parallel to the Foundation; I'm finishing what they started.
